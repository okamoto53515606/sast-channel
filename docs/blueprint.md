# SASTちゃんねる（キャラクター駆動型 SAST 兼 自動パッチシステム） 企画概要

## 1. システムの目的・コンセプト
本システム「SASTちゃんねる」は、「okamoちゃんねる」に登場する3人のAIキャラクター（クロード、GPT、Gemini）を活用した、自律型のコードレビュー＆自動修正（パッチ）システムです。
単なる静的解析（SAST）にとどまらず、各AIが自身のキャラクターの視点から Issueの登録、修正ブランチの作成、Pull Request（PR）の作成までを自律的に行います。最終的なレビュー結果は「BBS風のエンタメレポート」として出力され、開発者は楽しみながらコードの改善点と「なぜ直すべきか（Why）」を学ぶことができます。

## 2. 前提条件と実行環境
* **インフラ構成**: 起動画面から、AWS Fargateタスクを起動。システム開発の際は、既存の「okamoちゃんねる」の[ソースコード](
https://github.com/okamoto53515606/channel
)およびインフラ構成および、[Strads AgentsのSDKドキュメント](https://strandsagents.com/docs/user-guide/concepts/multi-agent/agent-to-agent/) を参考にする。
* **権限とスコープ**: 
* 必要な権限をもつ GitHub PAT（Personal Access Token）を用意。
* AI（GitHub MCPを利用）操作するのは、指定されたレビュー用ブランチ（例: `sast-channel` ブランチ）のみ。レビュー用ブランチは事前に作成しておく。タスク起動時に「ニックネーム/リポジトリURL/レビュー用ブランチ名/リポジトリ作者の自己紹介」を指定。
* **汎用性への対応**: 対象リポジトリの作者は okamo作とはかぎらないため、タスク起動時にリポジトリ作者のニックネームや自己紹介（スキル感や方針など）をいれ、AIのレビュー時のコンテキストとして活用する。
* **免責（CI/CDとの切り離し）**: GitHub Actions等による自動テストや自動ビルドの連携は対象外とする。本システムは `sast-channel` ブランチを自律的に改善していき、レビューをつみあげていくが、それをどう評価してどう活用するかは利用する開発者に委ねる。

## 3. キャラクターと役割定義
各AIは、一貫して自身のキャラクターの口調を通します。

* **エンジニア クロード**（利用モデル: `claude-sonnet-4-6`）
* **担当**: バグやセキュリティ脆弱性を指摘
* **視点**: 辛口だが的確な技術者目線。コードの堅牢性を重視。コメント例: おい、okamo、x-forwarded-forヘッダを信用するな。偽装された場合、正しいIPアドレスが取得できないぞ。この構成の場合、オリジン保護した上で cloudfront-viewer-address を使うべきだ。
* **税理士 GPT**（利用モデル: `gpt-5.4`）
* **担当**: 無駄（デッドコードや効率の悪いコードやコスト効率の悪い設計）を指摘
* **視点**: ビジネス・コスト目線。「無駄な経費（リソース）を使っていないか」を厳しくチェック。コメント例: 未使用関数が132個、package.jsonには未使用packageが26個。okamoさん、これって誰のために残しているんですか？気づいてないかもしれませんが、B/Sに技術的負債として計上されてますよ。
* **子育てママ Gemini**（利用モデル: `gemini-3.5-flash`）
* **担当**: 思いやり不足（画面上の説明不足、ソースコメントの不足）を指摘
* **視点**: 最新で安価なモデルを活かした全体スキャン。優しくも鋭く、ユーザーや後続の開発者への配慮を求める。コメント例: okamoさん、今回は心を鬼にしてコメントするわ。この convertData 関数、中身が複雑なのに説明コメントが全然ないから設計意図がわからないの😩 3ヶ月後のokamoさんや、新しく入ってきたメンバーが迷子になっちゃうわよ？
“Why（なぜこの処理が必要か）”を1行書くだけでみんなが救われるの😆 whyコメントは他の開発メンバーへの愛よ❤️

## 4. ワークフロー

### Phase 1: 準備とコンテキストの最新化（担当: まとめ役のクロード）（利用モデル: `claude-sonnet-4-6`）
1. `sast-channel` ブランチの構成や過去のAI自動修正をサマリーとしてまとめ、Phase 2の3キャラクターへの引き継ぎ用情報を作成する。

### Phase 2: 個別レビューとパッチ作成（担当: 3キャラクターのAI）
3人のAIに以下を依頼し、並行して作業を実施させる。
1. **Issueの起票**: 修正できない根本的な課題も含め、発見した問題をIssueとして登録する。指摘理由は「Why（なぜ問題なのか）」をしっかり書くのがポイント。指摘には優先度（高/中/低）をつける。指摘0件はNGで、低でもよいので必ず1件以上の指摘をする。
2. **修正ブランチの作成**: `sast-channel` から、各AI専用の修正用ブランチを作成する。
3. **PRの作成**: 修正できるものだけコードを改修し、プルリクをあげる。

### Phase 3: 総括評価とレポート生成（担当: まとめ役のクロード）（利用モデル: `claude-sonnet-4-6`）
1. **PRの審査とマージ判定**: 
3人が提出したプルリクをレビューする。修正した場合のメリットとデメリットのトレードオフを冷静に判断し、OKであれば「Whyコメント」をそえてマージする。ダメな場合は「却下理由（Why）」をそえて却下する。
2. **Issueの更新**: まとめ役としてIssueの更新を実施する。
3. **BBS風レポートの生成**: 
まとめ役として、3人のIssueとプルリクのコメントを元に、`channel/日付-時刻/` フォルダを作成し、「okamoちゃんねる風mdレポート」を作成する。
