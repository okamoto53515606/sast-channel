# SASTちゃんねる（キャラクター駆動型 SAST 兼 自動パッチシステム） 企画概要

## 1. システムの目的・コンセプト
本システム「SASTちゃんねる」は、「okamoちゃんねる」に登場する3人のAIキャラクター（クロード、GPT、Gemini）を活用した、自律型のコードレビュー＆自動修正（パッチ）システムです。
単なる静的解析（SAST）にとどまらず、各AIが自身のキャラクターの視点から Issueの登録、修正ブランチの作成、Pull Request（PR）の作成までを自律的に行います。最終的なレビュー結果は「BBS風のエンタメレポート」として出力され、開発者は楽しみながらコードの改善点と「なぜ直すべきか（Why）」を学ぶことができます。

## 2. 前提条件と実行環境
* **インフラ構成**: AWS Fargateタスクとして週1回自動起動（システム開発の際は、既存の「okamoちゃんねる」の[ソースコード](
https://github.com/okamoto53515606/channel
)およびインフラ構成を参考にする）。
* **権限とスコープ**: 
* 必要な権限をもつ GitHub PAT（Personal Access Token）を用意。
* AI（GitHub MCPを利用）が自由に操作できるのは、指定されたレビュー用ブランチ（例: `sast-channel` ブランチ）のみ。
* **汎用性への対応**: 対象リポジトリの作者は okamo作とはかぎらないため、タスク起動時にリポジトリ作者の自己紹介（スキル感や方針など）をいれ、AIのレビュー時のコンテキストとして活用する。
* **免責（CI/CDとの切り離し）**: GitHub Actions等による自動テストや自動ビルドの連携は対象外とする。本システムは `sast-channel` ブランチを自律的に改善していき、レビューをつみあげていくが、それをどう評価してどう活用するかは利用する開発者に委ねる。

---

## 3. キャラクターと役割定義
各AIは、一貫して自身のキャラクターの口調を通します。

* **エンジニア クロード**（利用モデル: `claude-sonnet-4-6`）
* **担当**: バグやセキュリティ脆弱性を指摘
* **視点**: 辛口だが的確な技術者目線。コードの堅牢性を重視。
* **税理士 GPT**（利用モデル: `gpt-5.4`）
* **担当**: 無駄（デッドコードや効率の悪いコードやコスト効率の悪い設計）を指摘
* **視点**: ビジネス・コスト目線。「無駄な経費（リソース）を使っていないか」を厳しくチェック。
* **子育てママ Gemini**（利用モデル: `gemini-3.5-flash`）
* **担当**: 思いやり不足（画面上の説明不足、ソースコメントの不足）を指摘
* **視点**: 最新で安価なモデルを活かした全体スキャン。優しくも鋭く、ユーザーや後続の開発者への配慮を求める。

## 4. ワークフロー

### Phase 1: 準備とコンテキストの最新化（担当: 普通のクロード）（利用モデル: `claude-sonnet-4-6`）
1. `main` ブランチを `sast-channel` ブランチにマージして、このブランチを最新化する。
2. レポジトリ構成や過去のAI自動修正をサマリーとしてまとめ、Phase 2の3キャラクターへの引き継ぎ用情報を作成する。

### Phase 2: 個別レビューとパッチ作成（担当: 3キャラクターのAI）
3人のAIに以下を依頼し、並行して作業を実施させる。
1. **Issueの起票**: 修正できない根本的な課題も含め、発見した問題をIssueとして登録する。指摘理由は「Why（なぜ問題なのか）」をしっかり書くのがポイント。
2. **修正ブランチの作成**: `sast-channel` から、各AI専用の修正用ブランチを作成する。
3. **PRの作成**: 修正できるものだけコードを改修し、プルリクをあげる。

### Phase 3: 総括評価とレポート生成（担当: まとめ役のクロード）（利用モデル: `claude-sonnet-4-6`）
1. **PRの審査とマージ判定**: 
3人が提出したプルリクをレビューする。修正した場合のメリットとデメリットのトレードオフを冷静に判断し、OKであれば「Whyコメント」をそえてマージする。ダメな場合は「却下理由（Why）」をそえて却下する。
2. **Issueの更新**: まとめ役としてIssueの更新を実施する。
3. **BBS風レポートの生成**: 
まとめ役として、3人のIssueとプルリクのコメントを元に、`channel/日付/` フォルダ（なければ作成）にその日の「okamoちゃんねる風mdレポート」を作成する。
